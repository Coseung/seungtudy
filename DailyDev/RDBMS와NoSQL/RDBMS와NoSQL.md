
## 동기 
RDBMS 를 찾아보다가 NoSQL이라는 것도 발견했다.

---
## RDBMS 특징

**정형화된 데이터**: 행과 열로 구성된 테이블 형식을 사용

**데이터 무결성**: 트랜잭션을 통해 데이터의 일관성을 강력하게 보장

**복잡한 쿼리**: SQL을 사용하며 조인 연산으로 여러 테이블 결합 가능

**예시**: MySQL, PostgreSQL, Oracle
![](https://velog.velcdn.com/images/dreamdp01/post/c1089def-893c-4f36-968c-a1f686aa2171/image.png)



---
## NoSQL 특징

**유연한 데이터**: 스키마가 없거나 동적임 (Key-Value, Document, Graph 등)

**확장성**: 수평적 확장(Scale-out)이 용이함

**빠른 속도**: 데이터 중복을 허용하며 응답 속도가 빠름

**예시**: MongoDB, Redis, Cassandra
![](https://velog.velcdn.com/images/dreamdp01/post/1fbf2b4a-8ec2-4982-a74f-c0115f6e24aa/image.png)

## 비교 및 선택 기준

무조건 관계형 데이터베이스인 **RDBMS**가 좋은 게 아니었다. 데이터 성격에 따라 명확하게 용도가 갈린다.

#### 1. RDBMS를 선택해야 할 때는 데이터의 정합성이 최우선일 때. 
예를 들어 전자상거래의 결제 시스템이나 재고 관리 시스템을 구축한다면 RDBMS가 적합하다. 돈이 오가는 문제는 트랜잭션 관리가 필수이기 때문이다. 또한 데이터 구조가 복잡하고 테이블 간의 관계가 명확할 때 유리하다.

#### 2. NoSQL을 선택해야 할 때는 데이터가 초대용량이거나 입출력 속도가 중요할 때. 
예를 들어 실시간 모바일 게임의 랭킹 정보나 IoT 센서 데이터 수집 같은 경우다. 이런 데이터는 구조가 자주 바뀌거나 유입량이 엄청난데, NoSQL은 수평적 확장이 쉬워서 이를 감당할 수 있다.

---
## 요약
데이터 안정성, 일관성 중요 (결제, 재고) -> RDBMS

대규모 데이터, 빠른 속도, 유연한 구조 중요 (게임 로그, IoT) -> NoSQL
